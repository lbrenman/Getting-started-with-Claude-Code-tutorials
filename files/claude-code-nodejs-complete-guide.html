1 · Why Claude Code beats browser chat

What fundamentally changes when Claude runs in your terminal

Browser Chat (old way)
  • Copy code in, paste code out manually
  • No knowledge of your project structure
  • Context lost when tab closes
  • One file or snippet at a time
  • Cannot run your server or tests
  • You integrate everything yourself
  • Re-explain conventions every session
Claude Code (new way)
  • Reads and writes files directly on disk
  • Sees your entire project tree
  • Persistent memory via CLAUDE.md
  • Multi-file edits in one task
  • Runs npm test, npm run dev, git commands
  • Shows diffs — you approve before any write
  • CLAUDE.md loaded automatically every session
Core mental shift: Stop thinking of Claude as a chatbot you consult. Think of it as a senior developer pair-programming in your terminal — it reads your files, writes code, runs tests, and iterates. You set direction and approve changes.
Three habits that make the biggest difference
  • 1Plan before you code. Use Plan Mode (Shift+Tab) to explore and design before any file is written. Anthropic's own guidance: "Letting Claude jump straight to coding can produce code that solves the wrong problem."
  • 2Write CLAUDE.md for every project. Claude's persistent memory — your stack, file layout, conventions, and npm scripts. Loads at the start of every session automatically.
  • 3Commit after each discrete task. Ask Claude to commit as it completes tasks. Small, reviewable diffs and clean rollback points make the whole workflow manageable.

2 · The planning workflow

Write markdown specs first — then implement one task at a time

Why plan first?
The biggest improvement over browser chat is writing a spec before any code is written. When Claude has a written plan to reference, it makes targeted changes, stays consistent across sessions, and produces code that fits your architecture from the start. The workflow produces four sequential markdown files — each feeds the next, and you review each before moving forward.
The four planning documents
1
docs/requirements.md — What are we building?
User stories, acceptance criteria, scope. Enter Plan Mode first (Shift+Tab) — Claude cannot write code in this mode. It interviews you with clarifying questions, then writes this file.
Enter Plan Mode (Shift+Tab) then use this prompt
I want to add [feature] to this project. You are in plan mode — do not write any code. Ask me clarifying questions about requirements, edge cases, and scope. Then write docs/requirements.md covering: what it does, who uses it, acceptance criteria, and what is out of scope.
2
docs/design.md — How will we build it?
Files to create or modify, data models, route shapes, middleware order. Claude reads requirements.md and your existing code. Still in Plan Mode — blueprint only.
Still in Plan Mode
Read docs/requirements.md and the current src/ structure. Write docs/design.md: which files will be created or changed, data models, route shapes or API endpoint definitions, and middleware needed. No code — blueprint only. Explain your reasoning for key decisions.
3
docs/tasks.md — In what order?
A numbered checkbox list. Each task touches at most 3 files, has a paired test task, and builds on the previous. You reorder or adjust before anything is coded. Each task = one commit.
Still in Plan Mode
Read docs/design.md. Create docs/tasks.md as a numbered checklist. Each task: touches at most 3 files, is independently testable, builds on the previous, and has a paired test task. Mark every item [ ] (not started).
4
Implement — one task at a time in Normal Mode
Exit Plan Mode (Shift+Tab). Claude implements Task 1, runs tests, commits, marks it done — then stops and waits for your review before Task 2.
Exit Plan Mode first (Shift+Tab back to Normal)
Read CLAUDE.md, docs/design.md, and docs/tasks.md. Implement Task 1 only. Run `npm test` after writing. Show me the results. If tests pass, mark Task 1 [x], commit with a conventional commit message, then stop and wait for my review before moving to Task 2.
One task at a time. Never ask Claude to implement the whole list at once. Small, reviewable diffs are the entire point of planning first.
Plan Mode, Normal Mode, Accept All
Shift+Tab → Plan Mode ⏸
Read-only. No file writes. Use for all three planning phases.
Shift+Tab → Normal Mode ▶
Asks permission per file write. Use for implementation. Review each diff before approving.
Shift+Tab → Accept All ⏵⏵
Auto-approves all writes. Use only for fully mechanical tasks — generating tests, reformatting.
/clear between tasks
Resets conversation. CLAUDE.md re-reads on next prompt. Use between unrelated tasks.
Context degrades above 70% usage. Use /compact proactively. Point Claude to docs files instead of re-explaining: "Read docs/design.md for the data model."

3 · CLAUDE.md — project memory

The file Claude reads at the start of every session, automatically

What CLAUDE.md is
CLAUDE.md sits at the project root and loads automatically at the start of every Claude Code session. It replaces everything you used to re-explain in browser chat — stack, file layout, conventions, npm scripts, and gotchas. Write it once; use it in every session forever. Keep it under 100 lines — every line costs context.
CLAUDE.md — master template (adapt for every project)
# Project: [Name] [One sentence: what this app does and who uses it.] ## Stack - Node.js 20 LTS, Express 4.x - [Templating: EJS | None — JSON API only] - [Database: PostgreSQL via pg | MongoDB via Mongoose] - [Auth: express-session + bcrypt | JWT (jsonwebtoken)] - Testing: Jest + Supertest - Dev tooling: nodemon, ESLint, dotenv ## Dev Commands ``` npm run dev # nodemon src/server.js (port 3000) npm test # jest --forceExit npm run lint # eslint src/ npm run lint:fix # eslint src/ --fix ``` ## Project Structure ``` src/ app.js # Express app — no listen() call here server.js # Entry point — binds PORT from env routes/ # One file per resource controllers/ # Business logic — routes stay thin middleware/ # asyncHandler.js, errorHandler.js, auth.js models/ # DB models or query functions views/ # EJS templates (web app only) public/ # Static assets (web app only) utils/ # Shared helpers tests/ # Mirrors src/ layout docs/ # Planning: requirements.md, design.md, tasks.md .devcontainer/ # devcontainer.json — Codespace config ``` ## Coding Standards - async/await everywhere — no raw Promises or callbacks - Named exports only — no default exports - Wrap ALL async route handlers with asyncHandler - Validate all req.body inputs before using them - Errors: throw new AppError("message", statusCode) - HTTP status codes always explicit - Secrets via process.env only — never hardcoded - Parameterized DB queries — never string interpolation ## Environment Variables See .env.example for all required variables. ## Commits Conventional commits: feat:, fix:, docs:, refactor:, test:, chore: Never commit .env. Commit after each completed task. ## Testing Supertest for route integration tests. Mock external services. Run npm test before every commit.

4 · Project structure & key files

What lives where, what gets committed, what never does

Full repo layout
my-project/ ├── CLAUDE.md # Claude's memory — always commit ├── .claude/ │ ├── settings.json # Tool permissions — commit │ └── commands/ # Custom slash commands (optional) ├── .devcontainer/ │ └── devcontainer.json # Codespace config — commit ├── docs/ # Planning docs — commit │ ├── requirements.md │ ├── design.md │ └── tasks.md ├── src/ # Application code ├── tests/ ├── README.md # With Codespace badge — commit ├── .env.example # Dummy values only — commit ├── .env # Real secrets — NEVER commit ├── .gitignore # Must cover .env — commit ├── .claudeignore # Keeps context lean (optional) └── package.json
The golden rule: .env is never committed. The three files that replace it are: .gitignore (blocks the commit), .env.example (documents what's needed), and GitHub Codespace Secrets (injects values automatically in a Codespace).
.gitignore — generate at scaffold time
Generate this on day one and it rarely changes. The most important entry is .env — everything else is standard Node.js housekeeping.
Claude Code prompt — run during scaffold

Include: - node_modules/, npm debug logs, .npm/, .node_repl_history - .env and .env.* — add a comment: "# Secrets — NEVER commit" - dist/, build/, out/, coverage/, .nyc_output/ - logs/, *.log, pids/, *.pid - .DS_Store, Thumbs.db, Desktop.ini - .vscode/settings.json (keep .vscode/extensions.json) - .claude/settings.local.json, CLAUDE.local.md
Verify it's working

.env.example — draft at scaffold, refresh before every push
Anyone who clones your repo or opens it in a Codespace needs to know which environment variables to set. .env.example is the only documentation for this. Never put real values in it — dummy placeholders and explanatory comments only.
Draft prompt — run during scaffold
Claude Code prompt
Create .env.example with safe dummy values for every process.env reference in this project. Add a one-line comment above each variable explaining what it does and where to get the value. Include a generation command for any secrets: node -e "console.log(require('crypto').randomBytes(32).toString('hex'))" Group related variables. Confirm .env is in .gitignore and .env.example is NOT in .gitignore.
Refresh prompt — run before every push to GitHub
Claude Code prompt — run this before git push
Scan every file in src/ for process.env references. Compare against .env.example. For any variable in the code but missing from .env.example: add it with a dummy value and a comment. For any variable in .env.example no longer in src/: remove it and tell me what was removed. Show me the final .env.example before writing it.
What a good .env.example looks like
# Server # Port the Express server listens on PORT=3000 # Use "development" locally, "production" in deploy NODE_ENV=development # Database # Full PostgreSQL connection string DATABASE_URL=postgres://myuser:mypassword@localhost:5432/myapp_dev # Authentication # Random 32-char string — generate with: # node -e "console.log(require('crypto').randomBytes(32).toString('hex'))" SESSION_SECRET=replace-with-a-random-32-char-string JWT_SECRET=replace-with-a-random-32-char-string
Make the refresh a habit. Every time you add a new process.env.SOMETHING, run the refresh prompt before pushing. It catches every gap automatically.
.claude/settings.json — tool permissions
{ "permissions": { "allow": [ "Bash(npm run *)", "Bash(npm test)", "Bash(node *)", "Bash(git status)", "Bash(git diff *)", "Bash(git log *)", "Bash(git add *)", "Bash(git commit *)" ], "deny": [ "Read(.env)", "Read(.env.*)", "Bash(rm -rf *)", "Bash(curl *)" ] } }
Lets Claude run npm scripts and git commands without interrupting you for permission, while blocking it from reading your secrets or running destructive commands.

5 · Codespace-ready devcontainer

The file that makes your repo run in Codespaces on first open

How Codespaces uses devcontainer.json
When someone opens your repo in GitHub Codespaces, it looks for .devcontainer/devcontainer.json. That file specifies the container image, VS Code extensions to auto-install, ports to forward, and commands to run after the container builds. Get this right and anyone gets a fully working dev environment — with your app already running — in under a minute.
postCreateCommand
Runs once when container is first built. Use for npm install.
postAttachCommand
Runs when VS Code connects. Use npm run dev to auto-start your server.
forwardPorts
Port 3000 for your app, 9229 for Node debugger.
onAutoForward
"openBrowser" for web apps (opens a tab). "notify" for APIs (shows a toast).
devcontainer.json — web app (Express + views)
// .devcontainer/devcontainer.json { "name": "Express Web App", "image": "mcr.microsoft.com/devcontainers/javascript-node:20", "customizations": { "vscode": { "extensions": [ "dbaeumer.vscode-eslint", "esbenp.prettier-vscode", "mikestead.dotenv", "christian-kohler.path-intellisense", "PKief.material-icon-theme" ], "settings": { "editor.formatOnSave": true, "editor.defaultFormatter": "esbenp.prettier-vscode" } } }, "forwardPorts": [3000, 9229], "portsAttributes": { "3000": { "label": "Web App", "onAutoForward": "openBrowser" }, "9229": { "label": "Node Debugger", "onAutoForward": "silent" } }, "postCreateCommand": "npm install", "postAttachCommand": "npm run dev" }
postAttachCommand: "npm run dev" is the magic line. When the Codespace opens, your server starts and a browser tab opens to port 3000 automatically — no terminal commands needed.
devcontainer.json — REST API (JSON only)
// .devcontainer/devcontainer.json { "name": "Express REST API", "image": "mcr.microsoft.com/devcontainers/javascript-node:20", "customizations": { "vscode": { "extensions": [ "dbaeumer.vscode-eslint", "esbenp.prettier-vscode", "mikestead.dotenv", "humao.rest-client", "PKief.material-icon-theme" ], "settings": { "editor.formatOnSave": true } } }, "forwardPorts": [3000, 9229], "portsAttributes": { "3000": { "label": "REST API", "onAutoForward": "notify" }, "9229": { "label": "Node Debugger", "onAutoForward": "silent" } }, "postCreateCommand": "npm install", "postAttachCommand": "npm run dev" }
For APIs use "notify" not "openBrowser" — a toast with the forwarded URL is more useful than a browser tab opening to a raw JSON endpoint. The REST Client extension lets you send requests from .http files directly in VS Code.

8 · Push to GitHub with README & Codespace badge

Why README comes last, how the badge works, and the pre-push checklist

Why README.md is generated after the push
The "Open in GitHub Codespaces" badge URL contains your GitHub username and repo name — so it can only be correct after the repo exists on GitHub. Generating the README locally before pushing means the badge URL will be wrong or use placeholder text. The correct sequence is: finish local dev and testing → push to GitHub → generate README.
This is the one file that intentionally comes last. Everything else — .gitignore, .env.example, devcontainer.json, CLAUDE.md — is generated at scaffold time.
The Codespace badge — how it works
The badge is a standard markdown image linked to https://codespaces.new/OWNER/REPO. GitHub resolves that URL to your repo's Codespace creation page. Anyone can spin up a Codespace in one click.
Badge markdown — replace OWNER and REPO
[![Open in GitHub Codespaces](https://github.com/codespaces/badge.svg)](https://codespaces.new/OWNER/REPO?quickstart=1)
Open in GitHub Codespaces
How it renders in a GitHub README
Always use ?quickstart=1. This shows a "Resume codespace" page if the user already has one open for this repo — rather than always creating a new one. Better UX for everyone.

9 · Daily session patterns

How to open, run, and close a productive Claude Code session

Opening a session
1
Launch Claude Code
cd my-project && claude
2
Orient Claude to where you left off
Session opener
Read CLAUDE.md and docs/tasks.md. Summarize which tasks are [x] complete, which is next [ ], and any blockers you can see in the code. Then we'll pick up from there.
Claude reads the task list and gives a status update. No re-explaining needed — this is the whole point of CLAUDE.md and the planning docs.
3
Work task-by-task or address a bug
"Implement the next unchecked task in tasks.md" — or pivot to a bug by pasting the full stack trace and asking Claude to diagnose before fixing.
Git workflow during a session
Ask Claude to propose a commit message
Task 3 is complete and tests pass. Review the changed files with `git diff`, then propose a conventional commit message. Show me the message — don't commit until I approve.
Or commit yourself
git add -p # Review changes interactively git commit -m "feat(auth): add requireAuth middleware" git push
Always review before committing. Claude Code writes directly to disk. Run git diff or git add -p so you see exactly what changed.
Debugging pattern
Diagnose before fix — always
POST /api/v1/posts returns 500. Stack trace: [paste full stack trace] Expected: 201 with the created post. Actual: 500 Internal Server Error. Read the relevant source files first. Tell me the root cause and why. Then propose a fix — wait for my approval before changing any files.
The "diagnose before fixing" pattern catches cases where Claude's first instinct is wrong. You can redirect before any file changes.
Context management
/compact at 70%
Compresses conversation to free space. Use proactively — quality degrades noticeably above 70%.
/clear between tasks
Full reset. CLAUDE.md re-reads on next message. Best between unrelated tasks or after long debugging sessions.
Point to docs
"Read docs/design.md" costs far less context than pasting the content into the chat yourself.
.claudeignore
Like .gitignore for Claude's context. Exclude node_modules, dist, coverage, *.min.js, large generated files.
Keeping CLAUDE.md current
Run this after significant feature additions
We've added features since CLAUDE.md was last updated. Read the current codebase and compare against CLAUDE.md. Update CLAUDE.md to reflect: new dependencies, changed folder structure, new environment variables, evolved conventions. Keep it under 100 lines.
Update CLAUDE.md whenever you add a dependency, env variable, change folder structure, or change a convention. It's the one file that makes every future session better.
